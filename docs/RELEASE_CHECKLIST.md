# Release and Squash Checklist

Use this checklist before creating a final squashed release commit/PR.

## 0. Tag policy (pre-v1 and v1)

- [ ] Pre-v1 feature validation happens on `vX.Y.Z-rc.N` tags first (dry-run release path).
- [ ] Stable `vX.Y.Z` tag is cut only after RC smoke tests pass on:
  - Android APK install + onboarding
  - Windows agent binary
  - Linux agent binary
- [ ] `publish-release.ps1` metadata and checklist in `docs/RELEASE_NOTES.md` are fully updated before publishing.

## 1. Working tree hygiene

- [ ] `git status --short` only shows expected changes.
- [ ] Local-only files are excluded (local SDK/config files, secrets).
- [ ] Generated artifacts are not staged.

## 2. Validation gates

- [ ] Agent tests pass:
  - `cd lumos-agent`
  - `go test ./...`
  - `go vet ./...`
- [ ] App checks pass:
  - `cd lumos_app`
  - `flutter analyze`
- [ ] If app UI changed, run at least one install/build path:
  - `flutter build apk` (or project build script)

## 3. Manual verification

- [ ] Verify agent web UI:
  - tab routing/deep links
  - scan presets and scan progress
  - safe mode confirmation + cooldown behavior
  - audit trail entries for local/relay actions
- [ ] Verify app flow:
  - onboarding
  - scan + add device via pairing token
  - action controls and health/capability display

## 4. Screenshot capture set

- [ ] Capture updated screenshots for:
  - agent dashboard (control panel)
  - scan tab with results/progress summary
  - settings tab (safe mode controls)
  - app onboarding and main device list
- [ ] Place screenshots in a stable path (for example `branding/screenshots/`).
- [ ] Confirm images do not expose private tokens/passwords/IPs unintentionally.

## 5. Commit grouping before squash

- [ ] Group commits by concern:
  - backend behavior/security
  - UI/UX changes
  - app-side changes
  - tests/docs
- [ ] Ensure each grouped commit message describes intent and scope.
- [ ] Keep unrelated formatting-only changes isolated.

## 6. Squash procedure

- [ ] Create a backup branch before rewriting:
  - `git branch backup/pre-squash-<date>`
- [ ] Squash to final cohesive commit set (or single release commit).
- [ ] Re-run validation gates after squash.
- [ ] Force-push only when required and confirmed:
  - `git push --force-with-lease origin <branch>`

## 7. Final release notes

- [ ] Summarize user-visible changes.
- [ ] Summarize security/hardening updates.
- [ ] Include test evidence and screenshot references.
- [ ] List known limitations and follow-up work items.

## 8. Release integrity

- [ ] `checksums.txt` is present in release assets.
- [ ] SBOM assets are present (`go-modules.json`, `flutter-packages.txt`).
- [ ] Build provenance attestations were generated by release workflow.
- [ ] If signing secrets are configured, checksum signature assets are uploaded.
